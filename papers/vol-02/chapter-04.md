# 第4章 統計/ML 分析用 Skill の設計原則

> **本章の到達目標**
> - vol-01 第7章の 6 要素（目的／入力条件／出力形式／成功条件／禁止事項／再現性条件）を、統計/ML Skill 向けに拡張できる

>
> **本章で扱わないこと**
> - CV の詳細設計（k-fold / grouped / time-series / nested）→ **第7章**
> - 具体的な scikit-learn API・コード実装 → 第5章以降
> - 事前分布・MCMC 診断の書き方 → 第10章・第12章
> - provenance フィールドの完全定義 → 付録A

## 4.1 なぜ「設計」から始めるのか（統計/ML 版の動機）

vol-01 第7章で導入した「Skill 仕様書 6 要素」は、**分析を実装する前に、日本語で目的と契約を書き下す**規律でした。統計/ML では、この規律の**重要性がさらに増します**：

- **「動くコード」の裏で、統計的には破綻している**——決定係数 0.95 の裏で、同一試料が train/test 両方に入っていた
- **「精度が良いモデル」の中身が説明できない**——共同研究者に「なぜこの元素が効いているのか」を聞かれて答えられない
- **「有意な差」を出すために、後から評価指標を変えた**——p 値 0.06 → 相関 r=0.7 に切り替えて「有意」と主張

これらはすべて **「成功の定義を曖昧なまま実装に入った」** ことが原因です。第4章では、実装に入る前に、**「何を成功とみなすか」** を厳密に書き下す方法を扱います。

> [!IMPORTANT]
> **本章の主題は「成功の定義」であって、「どう推定するか」ではありません**。CV の設計、ハイパーパラメータ探索、ネスト CV は第7章で扱います。ここで先に成功を定義しておかないと、第7章の CV 設計は「何を最適化するか」が空欄のまま始まってしまいます。

## 4.3 成功条件を書き下す：評価指標・目的関数・許容範囲

「成功条件」を統計/ML 用に書き下す際は、次の **3 点セット** を必ず含めます：

1. **評価指標**（何を測るか）
2. **目的関数**（学習時に最小化/最大化するもの）
3. **許容範囲**（どの値以上・以下なら成功か）

### 3 点セットが揃った成功条件の例

> ✅ **成功条件（回帰の例）**：
> - 評価指標：**RMSE（GPa 単位）**
> - 目的関数：MSE（学習時の損失）
> - 許容範囲：**5-fold CV の平均 RMSE ≤ 2.5 GPa、標準偏差 ≤ 0.5 GPa**。ただし各 fold の RMSE が 3.5 GPa を超えるものが 1 つでもあれば失敗

> ✅ **成功条件（分類の例）**：
> - 評価指標：**F1 スコア（macro 平均）**、副次的に混同行列
> - 目的関数：log loss
> - 許容範囲：**5-fold CV の平均 F1 ≥ 0.80、各クラスの再現率 ≥ 0.70**

### 3 点セットが揃っていない、悪い成功条件

> ❌ 「精度が良いこと」——**指標が未定義**
>
> ❌ 「精度 90% 以上」——**多クラス分類で 90% は「全部を最頻クラスに分類」でも達成可能**
>
> ❌ 「r² が高いこと」——**閾値がない**
>
> ❌ 「有意な相関があること」——**p 値の閾値、多重比較補正、事後選択の禁止が未定義**

### 主な評価指標と使い分け

| 指標 | 用途 | 注意点 |
|---|---|---|
| **RMSE / MAE** | 回帰、絶対誤差 | 単位が結果と同じ、外れ値の影響を RMSE は受けやすい |
| **R²（決定係数）** | 回帰、分散説明率 | データ範囲に依存、外挿では信頼できない |
| **MAPE** | 相対誤差重視の回帰 | 目的変数が 0 付近だと爆発、対数変換の検討を |
| **F1（macro / weighted）** | 不均衡分類 | クラス数が多い場合は macro、少数クラスの重みを尊重 |
| **ROC-AUC / PR-AUC** | 二値分類の閾値非依存評価 | クラス不均衡が強い場合は PR-AUC を優先 |
| **Log Loss / NLL** | 確率予測の質 | 予測確率のキャリブレーションを見るのに有用 |
| **物理的許容範囲**（例：予測値が非負） | 材料研究特有 | 統計指標が良くても物理的に破綻していれば失敗 |

### 目的関数と評価指標が違うことに注意

多くの機械学習モデルでは、**学習時に最小化する損失**と、**評価時に見る指標**が異なります：

- 回帰：学習は MSE、評価は RMSE や MAE
- 分類：学習は log loss、評価は F1 や ROC-AUC

**Skill 仕様書には両方を明記**します。混同すると、「学習は最適化されているのに評価指標では悪化」という事故（第14章）が起きます。

## 4.4 循環設計問題の統計版

第1章 §1.2 で予告した **循環設計問題の統計版** を、ここで正式に定義します。

### 定義

**エージェントが「評価指標も、モデル選択の実行も」両方を行うと、指標が「都合の良い方向」に選ばれるリスクがある** ——これが循環設計問題の統計版です。

### 具体例

```
ユーザー：「モデルを最適化してください」
エージェント：「RMSE で最適化しました。RMSE=2.3 GPa です」
ユーザー：「他の指標は？」
エージェント：「MAE=1.8, R²=0.72, MAPE=15% です」
ユーザー：「R² がもう少し欲しいですね」
エージェント：「R² を目的関数として再学習しました。R²=0.79 です」
```

この対話は一見自然ですが、**評価指標を最終結果に合わせて変えている**——これはデータリークと同種の科学的不正の入り口です。

### 回避策：Skill 契約に「指標の固定」を書き込む

Skill 仕様書に、以下を **④成功条件** として明記します：

> ✅ **④ 成功条件**（循環設計回避版）：
> - 評価指標は **RMSE（GPa）** に固定する。**Skill 実行後の評価指標変更は禁止**
> - 副次的指標として MAE, R², 予測範囲外率を報告するが、これらは合格判定には使わない
> - 合格判定の閾値（RMSE ≤ 2.5 GPa）は **Skill 設計時に固定** し、実行結果によって変更しない

そして、⑤禁止事項に：

> ✅ **⑤ 禁止事項**（循環設計版）：
> - **評価指標の後付け変更**（Skill 実行後に別指標を用いて「合格」と主張することを禁止）
> - **合格閾値の後付け緩和**（結果が悪かった時に閾値を下げることを禁止）
> - **モデル/特徴量の後付け追加**を、独立テストセットのスコアを見た後に行うこと（**test set への漏洩**）

### エージェント運用上のルール

- エージェントに「モデル選択を行ってください」と指示する前に、**Skill 契約に評価指標と閾値が書かれていることを確認**
- 実行後に「もっと良い指標はないか」と聞くこと自体は OK。ただし **その場で採用しない**——Skill 契約の改訂として、次のバージョンから使う
- 事後分析（探索的データ解析）と本評価は **別 Skill** として分ける

> [!WARNING]
> 循環設計問題は、AI エージェントを使わない従来の研究でも起きます。しかし **エージェントは高速に「探索」できてしまう**ため、リスクが大きく増幅します。第7章のネスト CV、第14章の p-hacking もこの延長線上にあります。

## 4.5 統計/ML Skill の禁止事項（vol-02 版）

vol-01 第7章の禁止事項（生データ上書き・外部送信・単位変換の暗黙実施）に加え、統計/ML Skill では**次の 6 項目**を必ず禁止事項として書き下します：

1. **データリーク（train/test 間の情報漏洩）**
   - 例：全データで正規化 → 分割、同一試料の反復測定を分割せず配置、時系列で未来を先に見る
   - 詳細は第7章
2. **特徴量への目的変数の混入**
   - 例：目的変数から派生した量（例：物性値を含む「材料コスト指標」）を特徴量に含める
3. **評価指標の後付け変更**（§4.4）
4. **合格閾値の後付け緩和**（§4.4）
5. **独立テストセットの複数回参照**
   - CV で選んだハイパーパラメータを、独立テストで確認するのは 1 回のみ。悪ければ Skill を再設計するのが正しい
6. **モデルの外挿域での予測を、事前警告なしに返すこと**
   - 例：学習データの温度範囲が 200-400 K のモデルに 600 K を入力しても、警告なしに数値を返す

これらを Skill の ⑤禁止事項 に必ず明記します。実装（fit / predict）の段階で **契約違反があれば fatal で拒否** する設計にします（vol-01 第8章の 3 段階を継承）。

## 4.6 再現性条件の拡張（vol-02 版）

vol-01 の再現性条件（`input_sha256` / `skill_version` / `run_datetime_utc` / `package_versions` / `random_seed`）に、統計/ML では次のフィールドを追加します。**完全定義は付録A**、ここでは概観のみ：

| フィールド | 意味 | 対応章 |
|---|---|---|
| `cv_scheme` | 分割方法（k-fold / grouped / time-series）、fold 数、グループ列、shuffle 設定、seed | 第7章 |
| `data_split` | 実際の train/val/test の index リストか fold ID | 第7章 |
| `model_config` | モデル種別（`RandomForestRegressor` 等）、ハイパーパラメータの完全リスト | 第5章 |
| `feature_columns` | 使用した特徴量の順序付きリスト（順序保存が重要） | 第5章 |
| `target_column` | 目的変数の列名と単位 | 第4章 |
| `metric_definition` | 評価指標の関数（`sklearn.metrics.mean_squared_error(..., squared=False)` の完全形） | 第4章 |
| `training_score` / `cv_score` | 学習・CV の各 fold スコアと集計値（平均・標準偏差） | 第5章 |
| **ベイズ用（第10章以降）** | `sampler_config` / `backend_config` / `posterior_artifact` / `diagnostics_summary` | 第10章 / 第12章 |

**再現性条件の中で最も抜けやすいのは `data_split` と `metric_definition`** です。前者は「同じ CV スコアを再現するために、どの試料が train でどの試料が test だったか」の情報、後者は「RMSE を計算するときに `squared=False` を指定していたか、`np.sqrt(mse)` を手計算していたか」の違いを吸収する情報です。

## 4.7 統計/ML Skill 仕様書テンプレート（vol-02 版）

以下が本章の主成果物です。第5章以降のすべての Skill 設計で、このテンプレートを起点にします。

```markdown
# Skill 仕様書：<Skill 名>

## ① 目的
- 何を予測/推定/分類するか、1 文で
- どんな判断に使うか（例：「合成条件の絞り込み」「装置校正の妥当性確認」）

## ② 入力条件
- **データ型**：spectrum / timeseries / image / pattern / tabular / multimodal のどれか
- **特徴量スキーマ**：列名・単位・型・値域・欠損許容度（vol-01 第8章の入出力スキーマ形式）
- **目的変数**：列名・単位・型・値域
- **サンプル数**：想定される最小・最大の n
- **学習/評価データの分離状態**：既に分かれているか、Skill が分割するか

## ③ 出力形式
- **学習済みモデル**：形式（joblib / pickle / ONNX 等）、保存パス
- **評価結果**：CV スコア（各 fold + 集計）、独立テストスコア（あれば）
- **予測（オプション）**：予測値、予測区間、外挿警告フラグ
- **provenance**：付録A 準拠

## ④ 成功条件（3 点セット）
- **評価指標**：<例：RMSE (GPa)>
- **目的関数**：<例：MSE>
- **許容範囲**：<例：5-fold CV 平均 RMSE ≤ 2.5、標準偏差 ≤ 0.5、各 fold ≤ 3.5>
- **物理制約チェック**：<例：予測値はすべて非負>

## ⑤ 禁止事項
- 生データ上書き、外部送信、単位変換の暗黙実施（vol-01 継承）
- **データリーク**：全データでの正規化、同一試料の分割違反、時系列の未来参照
- **特徴量への目的変数の混入**（派生量を含む）
- **評価指標の後付け変更**
- **合格閾値の後付け緩和**
- **独立テストの複数回参照**
- **外挿域での予測を警告なしに返すこと**

## ⑥ 再現性条件
- vol-01 継承：`input_sha256` / `skill_version` / `run_datetime_utc` / `package_versions` / `random_seed`
- vol-02 追加：
  - `cv_scheme`：分割方法・fold 数・グループ列・shuffle・seed
  - `data_split`：各 fold の train/test index
  - `model_config`：モデル種別・ハイパーパラメータ
  - `feature_columns`：特徴量順序
  - `target_column`：目的変数と単位
  - `metric_definition`：評価指標関数の完全指定
  - `training_score` / `cv_score`：学習・CV スコア
  - **（ベイズ Skill のみ）** `sampler_config`, `backend_config`, `posterior_artifact`, `diagnostics_summary`
```

### エージェントとの使い方

このテンプレートを Skill リポジトリの `SKILL.md` に置き、**エージェントは仕様書を読み込んでから実装セルを生成**します。仕様書に閾値・禁止事項が書かれていれば、エージェントの生成コードにも自動的に契約チェック・警告文が組み込まれます。

## 4.8 章末ワーク

以下のワークを行ってから第5章に進んでください。**この設計工程を省略すると、第5章以降のハンズオンで「何を成功とするか」の合意が取れなくなります**。

1. **自分の実験データで、1 つの回帰/分類タスクを設定**
   - 例：「XRD 20 点から硬度を予測する回帰」「Raman スペクトルから相同定する分類」
2. **仕様書テンプレートの ①②③④⑤⑥ を日本語で埋める**
   - 特に ④成功条件は **3 点セット**（指標・目的関数・許容範囲）を必ず含める
   - ⑤禁止事項は本章の 6 項目 + vol-01 の項目
3. **セルフレビュー**：以下を確認
   - [ ] ④成功条件に閾値が入っているか？
   - [ ] ⑤禁止事項に「評価指標の後付け変更」が含まれているか？
   - [ ] ⑥再現性条件に `data_split` と `metric_definition` が含まれているか？
   - [ ] エージェントが Skill を実行した際、「合格 or 不合格」が機械的に判定できるか？
4. **共同研究者・指導教員に見せる**：仕様書だけで、実装コードを見なくても「何をやろうとしているか」が伝わるかを確認

## 4.9 本章のまとめ

- 統計/ML Skill の設計は、vol-01 第7章の 6 要素を拡張する。中身は統計/ML 特有になる

- **再現性条件の拡張**：`cv_scheme` / `data_split` / `model_config` / `feature_columns` / `metric_definition` を追加。完全定義は付録A
- 次章（第5章）から、この仕様書テンプレートを使って **教師あり学習 Skill** を実装する
