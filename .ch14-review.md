### B-2: §14.2.1 randomization_integrity_check の 4-stage が Ch10 §10.5.3 canonical と一致しない

**箇所**: chapter-14.md L336-372（および §14.7 checklist L925）

**問題**: Ch14 §14.2.1 は「Ch10 §10.5.3 の 4-stage assignment_log detection」を検出契約として掲げているが、実際の Ch10 §10.5.3 の 4 stage と名称・意味が乖離：

| Ch10 §10.5.3 canonical (chapter-10.md L354-390) | Ch14 §14.2.1 (chapter-14.md L336-370) |
|---|---|
| `seed_match`（全レコード + header の seed 一致） | `seed_match_check` ≈ 同等 ✓ |
| `design_hash_match`（design_matrix_sha256 ↔ header） | **未実装** |
| `permutation_reproducibility`（Fisher-Yates byte-exact） | `assignment_order_match_check` + `execution_order_match_check`（順序差分の list、byte-exact を要求しない） |
| `execution_records_binding`（各 execution が design_matrix_sha256 / assignment_log_sha256 / run_id の 3 点を cite） | **未実装** |
| — | `assignment_log_header_check`（header 記録の有無のみ、実質的に何を verify しているか不明） |

**影響**: 「Ch10 §10.5.3 の 4-stage を全て含む」と §14.7 checklist は主張するが、canonical 4-stage の半分（`design_hash_match`, `execution_records_binding`）を捕捉しておらず、そもそも B-5（design matrix と assignment log の hash 連鎖）検出漏れが発生する。Ch10 が「seed 一致だけでは行入替を検出できない」と明示している問題点を、Ch14 の check が結局回避できていない。

**修正案**: Ch14 §14.2.1 の `automated_checks` を Ch10 §10.5.3 canonical と 1-to-1 対応させる。特に `design_hash_match` と `execution_records_binding` を追加。`assignment_order_match_check` と `execution_order_match_check` は `permutation_reproducibility` に統合し、`library: numpy_default_rng | pyDOE2` と `policy: byte_exact` を明示。

**根拠**: chapter-10.md L354-390 の canonical 4-stage 定義。Ch14 §14.7 checklist L925 の "Ch10 §10.5.3 の 4-stage detection を全て含む" 契約。

**根拠**: chapter-04.md L272-282（3 gate 表）と L336-361（facility_scope_escalation）。§14.7 checklist L949 は L4 gate が "Ch4 §4.6.2 の operational 版" と主張。

### B-5: `audit_manifest_sha256` の evidence_chain 組込みが提案止まり

**箇所**: chapter-14.md L806-808（§14.4 運用ポリシー）、L952（§14.7 checklist）

**問題**: §14.4 の「運用ポリシー」箇条書きで「Ch13 §13.4.4 の `evidence_chain_sha256_input_fields` に `audit_manifest_uri` と `audit_manifest_sha256` を追加すべき（Ch13 の canonical 拡張候補）」と述べる。§14.7 checklist でも「組み込むことが**議論されている**」で留まる。これは design flaw：
- Ch14 §14.5.1 の fatal `modify_audit_manifest_after_completion` は audit_manifest の immutability を要求
- しかし evidence_chain_sha256 の input field に組み込まれない限り、audit_manifest を差し替えても Ch13 §13.4.4 の RFC 8785 canonical hash は変化しない
- → **mechanical enforcement が存在しない fatal** になる（tampering しても検出できない）

**修正案**: 
- Ch13 §13.4.4 の `evidence_chain_sha256_input_fields` に `audit_manifest_uri` / `audit_manifest_sha256` を **back-register**（本文修正 + Ch13 §13.5 fatal 一覧に `modify_audit_manifest_input_fields_after_publication` を追加）
- §14.4 の「議論されている」を「Ch13 §13.4.4 に本章で back-register する」に格上げ
- §14.7 checklist を「組み込むことが議論されている」→「組み込まれている」に変更

**根拠**: chapter-13.md L748-770（`evidence_chain_sha256_input_fields` canonical）。§14.5.1 fatal `modify_audit_manifest_after_completion` の実効性要件。

### B-6: 16 パターンにネットワーク / 伝播系 Agentic 失敗が欠落

**箇所**: chapter-14.md 全体（特に §14.3）

**問題**: 6+4+6=16 の算術は成立するが、以下の Agentic 特有 pattern が unaddressed で、review context が明示的に懸念する「network（他プロジェクトへの伝播）」を実質カバーしていない：

1. **Facility standard silent inheritance**: `approved_facility_prior_specification`（Ch12 §12.8）や `approved_facility_response_surface_baseline`（Ch11）を promotion 済み後、下流プロジェクトが **audit_manifest re-run なしで inherit** する pattern。§14.3.6 は CATE broadcast 単発を扱うが、**継承的伝播**は扱っていない
2. **Sequential Bayesian DoE の feedback loop poisoning**: Ch12 §12.5 の Bayesian sequential design で、前 iteration の posterior が silent に次 prior として consumed される pattern。`prior_specification_provenance` の chain が途切れる
3. **Cross-project evidence_chain 参照の transitive integrity**: §14.3.6 `external_project_transfer` は「他プロジェクトへの流用」を扱うが、**参照される側（親 project）の evidence chain が invalidate された際、子 project 側の再検証 gate** が仕様化されていない

**影響**: これらは vol-04 中盤（Ch11-12）が facility 標準化を前提としている以上、Ch14 で捕捉しないと施設全体に silent に伝播する failure mode になる（Sculley 2015 の Hidden Technical Debt パターンの因果版）。

**修正案**: §14.3 に 3 パターン追加（あるいは §14.3.6 を分割）：
- `silent_facility_standard_inheritance_check`（下流 project が facility standard の hash pin 検証をスキップ）
- `sequential_bayesian_prior_chain_break_check`（Ch12 §12.8 prior provenance の chain integrity）
- `transitive_evidence_chain_invalidation_check`（親 evidence chain の hash 変化を子が検出）

16 → 19 に拡張、あるいは §14.3 を 9 パターンに再構成。§14.7 の "16 checks" 数字も更新。

**根拠**: chapter-04.md L336-354 `facility_scope_escalation.applies_to` の 8 種の promotion 対象（うち Ch11/Ch12/Ch13 promotion は伝播的）。chapter-12.md L1232 の "prior_family_change / prior_hyperparameter_change" back-registration。review context 冒頭の "network（他プロジェクトへの伝播）などの重要 pattern が漏れていないか" 懸念。

### N-8: audit_result_summary の集計 invariant が明示されない

**箇所**: chapter-14.md L777-782

**問題**: `pass_count + fail_count + not_applicable_count = 16` の invariant が canonical に書かれていない。監査 skill 実装で「invocation されなかった check」が missing で計上漏れになる可能性。

**修正案**: `audit_result_summary` に `total_checks_expected: 16` と `invariant: pass_count + fail_count + not_applicable_count == total_checks_expected` を追加。fatal `claim_audit_pass_without_running_all_16_checks` の operational detection にこの invariant を紐付ける。

**根拠**: §14.5.1 fatal L825 と §14.6 L900 が「16 checks 走らせずに pass 主張」を fatal 化しているが、mechanical detection の invariant がない。

### N-9: 演習 14.3 の `rfc8785` package の存在性

**箇所**: chapter-14.md L975-977

**問題**: `rfc8785` package を挙げているが、PyPI 実装（2024 時点）は `rfc8785`（Trail of Bits）と `json-canonicalize` 系が並存。片方だけ挙げると読者が version 差で hash 一致確認できない可能性。演習 14.3 は「hash 再計算のテストケース 3 個以上」を要求するので、実装非互換で fail する。

**修正案**: 演習 14.3 で library 選択の permitted list（`rfc8785` / `json-canonicalize` / Rust `serde_jcs` 経由）と、それぞれで同一 hash が得られることを **test case の 1 つに** 含める指示を追加。

**根拠**: RFC 8785 JCS は決定論的だが、Python 実装間の float 表現差で hash 齟齬が起きる known issue がある。
